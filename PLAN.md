# Summarizer Bot — план разработки

Телеграм-бот, который принимает ссылку (статья или YouTube-видео) и возвращает
краткий пересказ текста.

Документ — рабочий план для разработки совместно с ИИ-агентом. Держим его в
актуальном состоянии: по мере выполнения отмечаем шаги и фиксируем принятые решения.

---

## 1. Что делаем

**Вход:** сообщение в Telegram со ссылкой.
**Выход:** связный краткий пересказ текста по ссылке.

Поддерживаемые источники: веб-статьи и YouTube-видео (через субтитры/транскрипт).
Тип контента определяет извлекатель текста (rdrr) автоматически по URL.

---

## 2. Технологический стек

| Слой | Выбор | Почему |
|---|---|---|
| Язык | **TypeScript (Node.js 20+)** | rdrr — нативная TS-библиотека; один язык на весь проект; общие типы |
| Telegram | **grammY** | современный, отличная типизация, проще Telegraf, есть плагины (rate-limit, sessions) |
| Извлечение текста | **rdrr** (как библиотека) | `parse(url)` → markdown + метаданные; сам определяет YouTube/статью |
| LLM | **OpenAI-совместимый API** через `openai` SDK | провайдер задаётся `LLM_BASE_URL`+`LLM_API_KEY` (дефолт OpenRouter); меняем только `baseURL` |
| Оценка токенов | **эвристика символы÷3** | без зависимости; обрезка — редкий предохранитель |
| Конфиг | **zod** | валидация env-переменных на старте |
| Логи | **pino** | структурные логи |
| Хранилище | поначалу нет, далее **SQLite** | для MVP не нужно; позже — кэш по URL и история |
| Dev/Build | **tsx** (dev), `tsc`/`esbuild` (prod), **Docker** | |

> **Важно про rdrr:** это TypeScript-библиотека и CLI одновременно. Используем как
> библиотеку: `import { parse } from "rdrr"`. Не дёргаем через subprocess.

### Почему не Python
Можно было бы взять aiogram, но тогда rdrr пришлось бы вызывать как внешний CLI
(`npx rdrr <url>`), а Node всё равно поднимать. Лишний межъязыковой шов без выгоды.

## 3. Архитектура

Поток одного запроса:

```
Пользователь
   │  ссылка
   ▼
[grammY handler]  ── валидация URL, "печатает…", антиспам
   │
   ▼
[Extractor]  ── rdrr.parse(url) → { markdown, title, wordCount, type }
   │
   ├─ слишком длинно?  → чанкинг (map-reduce summary)
   ▼
[Summarizer]  ── OpenRouter chat completion (промпт + текст)
   │
   ▼
[Formatter]  ── markdown→Telegram (обрезка до 4096, разбивка)
   │
   ▼
Ответ пользователю
```

### Структура каталогов

Фактическая структура (`*.test.ts` лежат рядом с модулями, node:test):

```
src/
  bot.ts            # точка входа: grammY, middleware, heartbeat, graceful shutdown
  config.ts         # чтение env (zod): токены, модели, лимиты, allowlist…
  types.ts          # ExtractResult
  handlers/
    onLink.ts       # основной хендлер: URL → пайплайн → ответ + метрики
    onStart.ts      # /start, /help
  middleware/
    allowlist.ts    # фильтр по ALLOWED_USER_IDS (+ чистая isAllowed)
    rateLimit.ts    # лимит запросов/мин на пользователя
  core/
    extractor.ts    # обёртка над rdrr: таймаут, 3 ретрая, нормализация ошибок
    summarizer.ts   # вызовы OpenRouter, промпты, обрезка по токенам, fallback
    formatter.ts    # Telegram-HTML, шапка источника, разбивка ≤4096
  llm/
    openrouter.ts   # клиент (openai SDK с baseURL OpenRouter)
    prompts.ts      # системный и пользовательский промпты
  lib/
    url.ts          # извлечение и валидация ссылок
    logger.ts       # pino
    concurrency.ts  # лимитер очереди (MAX_CONCURRENCY)
    rateLimiter.ts  # скользящее окно
```

> `chunker.ts` (для map-reduce) пока не создан — см. Этап 2, выбран быстрый вариант.

Корень: `Dockerfile`, `docker-compose.yml`, `.dockerignore`, `tsconfig.json` /
`tsconfig.build.json` (сборка без тестов).

## 4. Ключевые проектные решения

Заложить с самого начала — это не «оптимизации на потом», а суть сервиса.

1. **Длинные тексты (главное).** Транскрипт часового видео не влезает в один запрос.
   Порог по токенам: коротко → один проход; длинно → **map-reduce** (суммаризировать
   чанки, затем суммаризировать саммари).

2. **Долгая обработка vs. UX.** Парсинг + LLM = 10–60 c. Сразу отвечаем
   «⏳ Обрабатываю…», шлём `sendChatAction("typing")`, не блокируем event loop.
   Очередь (хотя бы `p-queue` в памяти) защищает от заваливания ссылками.

3. **Ошибки rdrr.** Пейволы, видео без субтитров, 404, антибот-защита. Каждую
   категорию маппим в понятное сообщение пользователю, не в стек-трейс.

4. **Стоимость / злоупотребления.** OpenRouter платный: rate-limit на пользователя,
   опционально whitelist на старте, лимит по входным токенам.

5. **Выбор модели.** `MODEL` выносим в env, чтобы менять без передеплоя. Для
   суммаризации — баланс цена/качество у средних моделей.

- [x] `core/extractor.ts`: `rdrr.parse(url)` с таймаутом (поле текста — `content`)

### Этап 2 — длинные тексты
Выбран **быстрый вариант** вместо полного map-reduce: контекста современных моделей
обычно хватает на статью/транскрипт целиком, поэтому проще поднять лимит и честно
предупреждать об обрезке.
- [x] `MAX_INPUT_TOKENS` поднят до 200k (с запасом под контекст типичных моделей)
- [x] `capTokens` возвращает `{ text, truncated, keptPercent }`; при обрезке бот пишет
  «⚠️ Текст очень длинный — вошли первые ~N%», `truncated` идёт в метрики
- [ ] (Опц., если упрёмся) полный map-reduce: `chunker.ts` + суммаризация чанков →
  reduce, порог переключения. Минусы — много медленных LLM-вызовов и риск 429, см.
  обсуждение. Альтернатива попроще — «refine» (последовательное уточнение).
- **Готово:** тексты в пределах лимита суммаризируются целиком; длиннее — обрезаются
  с явным предупреждением пользователю.

### Этап 3 — устойчивость ✅
- [x] Категории ошибок rdrr → понятные сообщения пользователю (`ExtractError`)
- [x] Категории ошибок LLM → сообщения (`SummarizeError`: `unavailable`/`rate_limited`)
- [x] «typing»-индикатор и статус-сообщение «Обрабатываю…» (`onLink`)
- [x] Allowlist по Telegram ID (`ALLOWED_USER_IDS`, middleware до всех хендлеров)
- [x] Ретраи rdrr с backoff (3 попытки, кроме «пустого» результата); LLM 429 — повтор
  по `Retry-After`
- [x] Лимит конкурентности (`MAX_CONCURRENCY`, очередь в `lib/concurrency.ts`) +
  rate-limit на пользователя (`RATE_LIMIT_PER_MIN`, `lib/rateLimiter.ts` + middleware)
- **Готово:** бот не падает на битых ссылках, не блокируется спамом, не штормит 429
  при нескольких ссылках подряд, принимает команды только от разрешённых пользователей.

### Этап 4 — продакшн
- [x] `Dockerfile` (multi-stage, non-root, only prod-deps) + `.dockerignore`
- [x] Метрики: одна строка-лог `request` на запрос (`ok`, `model`, `parseMs`, `llmMs`,

- [x] Healthcheck (heartbeat-файл): бот обновляет `HEARTBEAT_FILE`, пока опрашивает

- [ ] Polling для старта; webhook — когда понадобится масштаб
- [ ] **При добавлении webhook — переделать healthcheck на HTTP `/health`** (вебхук и так
  поднимает HTTP-сервер; heartbeat-файл тогда убрать)
- [ ] (Опц.) SQLite-кэш по URL — не пересуммаризировать одинаковые ссылки
- **Готово, когда:** сервис собирается в образ и крутится в проде.

## 7. Принятые решения

### Модель — выбирается пользователем (дефолтов нет)
Бот для личного использования на бесплатном тиере. `MODEL` (обязателен) и
`MODEL_FALLBACK` (опционален, пробуется при ошибке/429) задаются в `.env` под выбранного
провайдера. **Встроенных дефолтов нет** — качество бесплатных моделей сильно разнится,
конкретную не навязываем.

- Бесплатные модели подбираются под себя; список текучий, проверяй живой:
  ```sh
  curl -s https://openrouter.ai/api/v1/models | \
    jq -r '.data[] | select(.id | endswith(":free")) | .id'
  ```
- Бот обрабатывает оба класса сбоев free-тиера: `SummarizeError` kind `unavailable`
  (404 — слаг протух/удалён) и `rate_limited` (429 — перегрузка, повтор по `Retry-After`).
  Если выбранная модель стабильно отдаёт 429 — поставь другой слаг или платный endpoint.

> **Влияние на план:** контекста современных моделей обычно хватает на статьи и
> транскрипты целиком, поэтому **Этап 2 (map-reduce) понижен в приоритете** — для MVP
> хватает одного прохода + обрезки по `MAX_INPUT_TOKENS`. Чанкинг — только если упрёмся.

### Язык и формат саммари
- **Язык: всегда русский**, независимо от языка оригинала.
- **Формат:**
  ```
  🔑 TL;DR — 2–3 предложения

  <жирный заголовок блока>
  • тезис
  • тезис

  <жирный заголовок блока>
  • …
  ```
- **Telegram не рендерит markdown-заголовки (`#`).** Поэтому:
  - parse mode — **HTML** (надёжнее MarkdownV2, без адского экранирования);
  - «заголовки блоков» — жирным (`<b>…</b>`), списки — `•`/`-`;
  - `formatter.ts` обязан экранировать `<`, `>`, `&` в тексте от LLM.

### Кэш
В MVP не делаем. Остаётся опциональным пунктом Этапа 4 на будущее.
