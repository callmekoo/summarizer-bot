# Summarizer Bot — план разработки

Телеграм-бот, который принимает ссылку (статья или YouTube-видео) и возвращает
краткий пересказ текста.

Документ — рабочий план для разработки совместно с ИИ-агентом. Держим его в
актуальном состоянии: по мере выполнения отмечаем шаги и фиксируем принятые решения.

---

## 1. Что делаем

**Вход:** сообщение в Telegram со ссылкой.
**Выход:** связный краткий пересказ текста по ссылке.

Поддерживаемые источники: веб-статьи и YouTube-видео (через субтитры/транскрипт).
Тип контента определяет извлекатель текста (rdrr) автоматически по URL.

---

## 2. Технологический стек

| Слой | Выбор | Почему |
|---|---|---|
| Язык | **TypeScript (Node.js 20+)** | rdrr — нативная TS-библиотека; один язык на весь проект; общие типы |
| Telegram | **grammY** | современный, отличная типизация, проще Telegraf, есть плагины (rate-limit, sessions) |
| Извлечение текста | **rdrr** (как библиотека) | `parse(url)` → markdown + метаданные; сам определяет YouTube/статью |
| LLM | **OpenRouter** через `openai` SDK | OpenRouter OpenAI-совместим: берём официальный SDK и меняем `baseURL` |
| Подсчёт токенов | **gpt-tokenizer** | для чанкинга длинных текстов |
| Конфиг | **zod** | валидация env-переменных на старте |
| Логи | **pino** | структурные логи |
| Хранилище | поначалу нет, далее **SQLite** | для MVP не нужно; позже — кэш по URL и история |
| Dev/Build | **tsx** (dev), `tsc`/`esbuild` (prod), **Docker** | |

> **Важно про rdrr:** это TypeScript-библиотека и CLI одновременно. Используем как
> библиотеку: `import { parse } from "rdrr"`. Не дёргаем через subprocess.

### Почему не Python
Можно было бы взять aiogram, но тогда rdrr пришлось бы вызывать как внешний CLI
(`npx rdrr <url>`), а Node всё равно поднимать. Лишний межъязыковой шов без выгоды.

## 3. Архитектура

Поток одного запроса:

```
Пользователь
   │  ссылка
   ▼
[grammY handler]  ── валидация URL, "печатает…", антиспам
   │
   ▼
[Extractor]  ── rdrr.parse(url) → { markdown, title, wordCount, type }
   │
   ├─ слишком длинно?  → чанкинг (map-reduce summary)
   ▼
[Summarizer]  ── OpenRouter chat completion (промпт + текст)
   │
   ▼
[Formatter]  ── markdown→Telegram (обрезка до 4096, разбивка)
   │
   ▼
Ответ пользователю
```

### Структура каталогов

```
src/
  bot.ts            # точка входа: grammY, регистрация хендлеров, graceful shutdown
  config.ts         # чтение env (zod): BOT_TOKEN, OPENROUTER_API_KEY, MODEL…
  handlers/
    onLink.ts       # основной хендлер: парсит URL, оркестрирует пайплайн
    onStart.ts      # /start, /help
  core/
    extractor.ts    # обёртка над rdrr: ретраи, таймаут, нормализация ошибок
    summarizer.ts   # вызовы OpenRouter, промпты, map-reduce для длинных текстов
    chunker.ts      # разбивка текста по токенам
    formatter.ts    # подготовка ответа под лимиты Telegram
  llm/
    openrouter.ts   # клиент (openai SDK с baseURL OpenRouter)
    prompts.ts      # системный и пользовательский промпты
  lib/
    url.ts          # извлечение и валидация ссылок
    logger.ts       # pino
  types.ts
```

## 4. Ключевые проектные решения

Заложить с самого начала — это не «оптимизации на потом», а суть сервиса.

1. **Длинные тексты (главное).** Транскрипт часового видео не влезает в один запрос.
   Порог по токенам: коротко → один проход; длинно → **map-reduce** (суммаризировать
   чанки, затем суммаризировать саммари).

2. **Долгая обработка vs. UX.** Парсинг + LLM = 10–60 c. Сразу отвечаем
   «⏳ Обрабатываю…», шлём `sendChatAction("typing")`, не блокируем event loop.
   Очередь (хотя бы `p-queue` в памяти) защищает от заваливания ссылками.

3. **Ошибки rdrr.** Пейволы, видео без субтитров, 404, антибот-защита. Каждую
   категорию маппим в понятное сообщение пользователю, не в стек-трейс.

4. **Стоимость / злоупотребления.** OpenRouter платный: rate-limit на пользователя,
   опционально whitelist на старте, лимит по входным токенам.

5. **Выбор модели.** `MODEL` выносим в env, чтобы менять без передеплоя. Для
   суммаризации — баланс цена/качество у средних моделей.

## 7. Принятые решения

### Модель — бесплатный тиер OpenRouter
Бот для личного использования. При росте нагрузки — переход на платный API (меняется
только `MODEL`/`MODEL_FALLBACK`, без передеплоя логики).

- **Основная:** `nvidia/nemotron-3-super-120b-a12b:free` — контекст 1M, ~14с на запрос,
  хороший русский. Хостится на инфраструктуре NVIDIA.
- **Фолбэк:** `nvidia/nemotron-3-nano-30b-a3b:free` — контекст 256k, ~1.5с, быстрый и
  надёжный (мелкие глитчи в тексте). Тоже NVIDIA.
- **Почему Nemotron, а не llama/qwen/gemma/deepseek.** Бесплатные слаги от
  Meta/Qwen/Google/DeepSeek роутятся через провайдера **Venice**, чей free-пул стабильно
  отдаёт **429** (перегружен), даже когда наш собственный лимит не выбран
  (`is_free_tier: true`, `usage: 0`). NVIDIA-модели идут через свободный пул и реально
  отвечают. Докупка кредитов это не лечит — free-трафик всё равно идёт через тот же
  затыканный пул; лечит только платный endpoint (~$0.001/пересказ).
- **Прежние дефолты (`gemini-2.0-flash-exp:free`, `deepseek-chat:free`) протухли** —
  OpenRouter их удалил (404). Урок: `:free`-слаги текучие, проверяй список:
  ```sh
  curl -s https://openrouter.ai/api/v1/models | \
    jq -r '.data[] | select(.id | endswith(":free")) | .id'
  ```
- Бот сам обрабатывает оба класса сбоев: `SummarizeError` kind `unavailable` (404,
  протухший слаг) и `rate_limited` (429, перегрузка — один повтор по `Retry-After`).

> **Влияние на план:** контекста 1M/256k хватает на статьи и транскрипты целиком,
> поэтому **Этап 2 (map-reduce) понижен в приоритете** — для MVP хватает одного прохода
> + обрезки по `MAX_INPUT_TOKENS`. Чанкинг добавляем, только если упрёмся в лимит.

### Язык и формат саммари
- **Язык: всегда русский**, независимо от языка оригинала.
- **Формат:**
  ```
  🔑 TL;DR — 2–3 предложения

  <жирный заголовок блока>
  • тезис
  • тезис

  <жирный заголовок блока>
  • …
  ```
- **Telegram не рендерит markdown-заголовки (`#`).** Поэтому:
  - parse mode — **HTML** (надёжнее MarkdownV2, без адского экранирования);
  - «заголовки блоков» — жирным (`<b>…</b>`), списки — `•`/`-`;
  - `formatter.ts` обязан экранировать `<`, `>`, `&` в тексте от LLM.

### Кэш
В MVP не делаем. Остаётся опциональным пунктом Этапа 4 на будущее.
